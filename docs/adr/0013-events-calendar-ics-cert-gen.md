# ADR-0013: Events — custom calendar, ics generation, pdf-lib cert overlay

**Status:** Accepted
**Date:** 2026-05-13
**Deciders:** Frontend lead, Supabase lead

## Context

The events feature must support: a calendar view with recurring events, listing tabs, an admin authoring flow with banner + certificate upload, attendance marking, and per-attendee cert generation + email delivery. Multiple library and architecture choices needed locking in before implementation.

## Decision

### Custom calendar component, no third-party calendar UI
We build the month-grid calendar from scratch as `frontend/components/events/EventCalendar.tsx` using a 7×6 CSS grid. Reasons:

* **Theme fidelity.** Every off-the-shelf calendar (FullCalendar, react-big-calendar, react-calendar) ships heavy CSS that fights Codemo tokens (`var(--bg-surface)`, `var(--accent-primary)`, etc.). Either we override their CSS — fragile — or we accept their look — wrong.
* **Bundle weight.** FullCalendar core + interaction + daygrid is ~200 KB minified gzipped. Our custom implementation is well under 10 KB.
* **Feature surface.** We need exactly: month navigation, category dots per day, today / selected highlighting, day-click handler. Building those from primitives is a single-evening job.

The form-side date picker uses native `<input type="datetime-local">` styled via the `codemo-input` utility for v1. Reusing the month-grid calendar as a date picker is tracked for the next admin sprint — the trade-off note is in [docs/ADMIN.md](../ADMIN.md#event-management).

### `rrule` (RFC 5545) for recurrence expansion in the browser
The admin form composes a small subset of RRULE strings (`FREQ=YEARLY;BYMONTH=…;BYMONTHDAY=…` etc.) and stores them on `events.recurrence_rule`. The calendar component expands them client-side via `rrule.between(rangeStart, rangeEnd, true)` for the visible month. Reasons:

* RFC 5545 is the lingua franca — same string flows into the .ics export.
* Client-side expansion keeps the server simple: the admin writes one row, the browser materialises the visible occurrences. No CRON job, no nightly materialisation table.
* Per-instance overrides live in `event_recurring_instances` (admin-only, override JSONB) for the rare case an admin needs to customise a single occurrence.

### `ics` library for client-side .ics download
The Apple / Outlook / Other "Add to Calendar" buttons trigger an `ics.createEvent()` call and download the resulting file with `Blob` + `URL.createObjectURL`. Reasons:

* No round-trip — the file generates instantly.
* Tiny library (~10 KB).
* Same `RRULE` value flows from the database into the .ics so recurrence carries.

Google Calendar uses its `calendar.google.com/calendar/render?action=TEMPLATE` URL scheme directly (no library — just URLSearchParams).

### `pdf-lib` for certificate overlay (Edge Function side)
The `generate-cert` Edge Function downloads the admin-uploaded template (PDF or PNG/JPEG) from Storage, composes a single PDF page via `pdf-lib`, draws the attendee name + event title + formatted date centred over the template, uploads the result, and signs a URL for 7 years. Reasons:

* `pdf-lib` is pure TypeScript, runs on Deno via the npm CDN (`https://esm.sh/pdf-lib`). No native binaries, no FFI, no Sharp.
* We considered `@napi-rs/canvas` for true canvas rendering. It is faster but pulls a native binary and has historically broken on Supabase Edge runtimes during version bumps. PDF-lib is portable and stable.
* The trade-off: text positions are fixed to centred bands at 55% / 42% / 32% of page height. Templates that include `{name} {event} {date}` placeholder blocks can be designed around those positions; finer positioning (per-template anchor coordinates) is a v2 enhancement.

### Storage layout: one bucket, three folders
`event-assets/banners/`, `event-assets/cert-templates/`, `event-assets/generated-certs/{event_id}/{user_id}.pdf`. Single private bucket keeps RLS simple — admins write everywhere, authenticated users read banners (signed URLs are generated by the server anyway), each user reads only their own cert. The folder-scoped read policy uses `(storage.foldername(name))[1]` and `split_part(name, '/', 3)`.

### No per-event RPC — direct PostgREST writes
Admin event create + update go through the standard `supabase.from('events').insert/update()` calls, gated by the existing JWT-role RLS policy added in `20260512000000_admin_panel.sql`. No `admin-create-event` Edge Function. Mutations that need privileged side effects (`mark-attendance`, `generate-cert`, `send-cert-email`) get Edge Functions because they cross trust boundaries (touch other users' rows, generate cross-user assets, call SMTP).

## Consequences

**Positive**
* Bundle stays light — custom calendar + `rrule` + `ics` total well under 50 KB gzipped vs ~250 KB for FullCalendar + ical.js.
* One source of truth for recurrence (`RRULE` string) feeds both the calendar grid and the .ics export.
* Cert generation does not depend on native binaries.

**Negative / known trade-offs**
* Our custom calendar lacks week + day views. Acceptable — the spec only requires month view.
* Cert text positioning is fixed at three centred bands. A future v2 should let templates define anchor coordinates in metadata.
* Native `datetime-local` inputs in the admin form do not match the user-facing `EventCalendar` visually. Reuse is queued.

## Alternatives considered

* **FullCalendar** — too heavy, theme override is a battle.
* **react-big-calendar** — same plus opinionated event rendering.
* **`@napi-rs/canvas` for cert generation** — faster but native, breaks across Edge runtime version bumps.
* **CRON-materialised recurring events** — a nightly job to expand `RRULE` into rows. Simpler to query but adds infra and gets stale on admin edits.
